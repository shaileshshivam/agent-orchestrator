PR #403 Design Iteration

Formalized Feedback Pipeline

This explainer aligns with docs/design/feedback-routing-and-followup-design.md and formalizes the report -> issue -> agent-session -> PR architecture with fork-aware execution.

What Changed in Design

  • Formalized end-to-end pipeline instead of only routing concepts.
  • Added explicit contracts for session spawn, target selection, and PR linking.
  • Defined idempotency + retry semantics as first-class behavior.
  • Added governance hooks for fork-owner policy control.

Core Design Decisions

  • Routing mode remains exclusive: local OR scm.
  • Orchestrator owns all side effects and retries.
  • Optional subagent skill can recommend decisions but cannot mutate SCM.
  • Privacy guardrails remain out-of-scope for this design iteration.

Formal Pipeline

1. Report Capture: validate feedback payload, compute dedupe key.
2. Issue Resolution: find/create/comment issue via dedupe markers.
3. Follow-up Planning: choose issue-only vs issue+PR vs issue+fork.
4. Execution: direct SCM operations or spawn agent session for code changes.
5. Linking + Journal: update issue/PR/fork references and operation status.

1) Trigger Conditions

  • Valid report + mode=scm + threshold met.
  • self_blocking_now=false -> issue-only.
  • self_blocking_now=true -> actionable path (PR/fork/session).

2) Session Spawn Contract

  • Requires existing issue URL and selected target repo.
  • Inputs include report ID, dedupe key, fork context, intent.
  • Outputs include session ID, branch ref, optional PR URL.

3) Target Selection

  • Prefer upstream when writable and policy allows.
  • Fall back to fork when upstream is blocked and fork allowed.
  • If neither allowed, downgrade to issue-only and record block reason.

4) PR Create/Link Requirements

  • Issue must exist and be linked in PR body.
  • Use markers for dedupe traceability.
  • Re-link existing PR when dedupe key already has one.

5) Idempotency + Retry

  • dedupeKey identifies issue-level identity.
  • operationKey scopes mutation retries.
  • All writes are find-or-create; retries are bounded with backoff.

6) Governance Hooks

  • Policy checks for issue, fork, PR, and session spawn.
  • Per-fork-owner allow/deny and approval controls.
  • Denied operations are non-mutating and journaled with reason.

Implementation Guidance (Next PRs)

  1. Router + issue resolver + publish journal.
  2. Target resolver + follow-up executor (direct SCM + session path).
  3. Fork-aware PR linking and governance hook enforcement.

Design Review Checklist

  • Is each decision deterministic when subagent is absent?
  • Can every stage be retried safely without duplicate side effects?
  • Are fork-owner policies enforceable per mutation type?
  • Is the issue->session->PR lineage auditable end to end?

Design-only update: this page formalizes architecture and contracts; behavior changes are intentionally deferred to follow-up implementation PRs.
