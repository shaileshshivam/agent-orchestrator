PR #403

Feedback Tools v1 Explainer

This page explains what the PR does, why it exists, what feedback is already addressed, and the current high-level and low-level design.

What This PR Does

  • Adds explicit tool contracts for bug_report and improvement_suggestion.
  • Enforces required fields: title, body, evidence, session, source, confidence.
  • Persists reports in structured key=value files under feedback-reports/.
  • Generates deterministic dedupe keys from normalized payload content.
  • Exports APIs from @composio/ao-core and documents usage.

Why It Exists

  • Enable decentralized self-improvement signal capture directly from agent sessions.
  • Provide machine-validated contracts (not ad-hoc blobs) for consistent ingestion.
  • Keep storage human-inspectable and aligned with AO's existing persistence style.
  • Allow lightweight duplicate suppression via stable dedupe keys.

High-Level Design

  • Two explicit tool contracts share one strict schema shape.
  • Validation and dedupe generation happen at write time.
  • Persistence is file-based, atomic, and project-scoped.
  • Read path is fail-soft: invalid single file does not poison full listing.

Low-Level Design

  • packages/core/src/feedback-tools.ts: schemas, dedupe, store, serialize/parse flow.
  • packages/core/src/key-value.ts: shared key=value parser utility.
  • packages/core/src/metadata.ts: now reuses shared parser utility.
  • packages/core/src/__tests__/feedback-tools.test.ts: required-field, confidence, dedupe, persistence, corruption resilience tests.
  • packages/core/src/paths.ts: getFeedbackReportsDir() path helper.

Scope intentionally excludes fork bootstrap and report->issue automation (per issue #399 constraints).
